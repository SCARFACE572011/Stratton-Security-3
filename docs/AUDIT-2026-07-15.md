# Site Audit — 2026-07-15

Full look-and-feel + SEO audit of the live site (https://stratton-v3.vercel.app) and codebase.
Method: 6 parallel audit agents (technical SEO, local SEO/content, performance, accessibility WCAG 2.2 AA, live-browser visual/UX review at 1440px + 390px, conversion/copy), followed by independent adversarial verification of every critical/high finding against the actual code and live site. **89 findings, 0 refuted.** Every critical/high item below was independently reproduced (verdict noted).

Counts: 7 critical · 23 high · 38 medium · 21 low.

## Dimension summaries

- **Technical SEO:** Solid technical-SEO foundation: all 48 sitemap URLs return 200 with no redirects, trailing slashes 308 to canonical form, garbage URLs return real 404s with a custom not-found page, robots.txt/sitemap.xml render from code as expected, favicons/apple-icon/theme-color are present, and every route has a canonical. However, three og:image endpoints (home, /about, /contact) serve zero-byte PNGs due to Satori render crashes; nearly every page title double-appends the brand via the root title template; robots.txt blocks /_next/ (all JS/CSS) while ~66 elements ship with inline opacity:0; and the live vercel.app host is fully indexable with canonicals pointing at the old Squarespace site. Schema has a self-serving aggregateRating/Review policy problem, a 404ing logo URL, and a geo/street-address contradiction; GA4 and Search Console emit nothing in production. Minor: meta keywords tag is emitted (obsolete, harmless) and FAQPage markup is valid but FAQ rich results are no longer shown for commercial sites.
- **Local SEO & Content:** Technical hygiene is genuinely strong — one unique H1 per page, canonicals, per-page BreadcrumbList/Service/FAQPage schema, clean semantic slugs, complete sitemap, and four substantive long-form guides — but the site systematically under-targets the commercial queries LA security buyers actually type: the homepage H1 is a slogan, service/industry H1s and titles omit 'security guards' and geo modifiers, and every detail page ships a doubled '| Stratton Security Group | Stratton Security Group' title suffix. The most urgent issues are a live NAP conflict (two different office addresses, with schema geo pointing at the old one), thin templated service-area pages that risk doorway-page treatment, and missing high-intent pages (fire watch, cost guide, major cities like Long Beach/Inglewood). NAP phone/email/license are otherwise perfectly consistent site-wide, and FAQ coverage of real buyer questions (pricing, contracts, switching vendors, deployment speed) is among the best parts of the site.
- **Performance:** PageSpeed Insights API was unavailable this run (429 RESOURCE_EXHAUSTED on the shared anonymous quota, after the initial calls plus two spaced retries), so lab scores come from the fallback: curl timing, live-HTML analysis, and byte-level bundle measurement. The server/CDN layer is excellent — TTFB 83-170ms, pages prerendered (x-nextjs-prerender: 1, x-vercel-cache: HIT), /_next/static immutable, fonts via next/font with display:swap, inner-page hero images correctly priority-preloaded with responsive srcsets. The problems are almost entirely self-inflicted client-side theatrics: a hard-coded 3.25s full-screen preloader on every page entry (SSR-rendered, so it also permanently covers content when JS fails), a hero whose LCP text is server-rendered invisible (translateY(115%)/opacity:0 on 63 elements) until 291KB gzip (943KB raw, measured across 17 chunks) of JS hydrates, a 139KB PNG logo preloaded on every page for a 44-120px slot, 19 below-fold client logos (188KB) eagerly preloaded in <head>, and an unconditional Lenis rAF loop — together these will put throttled mobile LCP in the 5-7s range and fail Core Web Vitals despite a ~100ms TTFB. Fixing the top four findings (preloader gating, visible SSR hero, LazyMotion + server-component sections, logo asset optimization) is mostly quick-to-moderate work and should move the mobile performance score dramatically before launch.
- **Accessibility:** The site has a genuinely strong accessibility baseline — exemplary FAQ accordion ARIA, correctly hidden honeypots, good alt text, inert off-screen carousel slides, and most Framer Motion gated behind useReducedMotion plus a global CSS reduced-motion kill switch. The dominant systemic defect is that the dark-theme override layer in globals.css (lines 660-735) remaps only exact base utility classes, so the global :focus-visible outline (#1a3a6b, 1.3-1.7:1 on dark surfaces), form error text (text-steel-700, 1.9:1), and hover-variant colors all fall below WCAG thresholds on the now-dark site. The second cluster is focus/keyboard management: no focus trap or Escape in the mobile drawer, a focusable button inside the aria-hidden preloader, a homepage-only skip link, and no focus handling between contact-form steps.
- **Conversion & Copy:** The conversion skeleton is genuinely strong — tel: links everywhere, a repeated free-assessment CTA block on nearly every page, honest specific FAQs (pricing, 72-hour deployment, switching vendors), good form success states with response-time promises, and clean professional copy with no typos or lorem. But the site undercuts itself at the exact top of the funnel: a 3.3-second fake 'secure session' preloader gates every landing, the homepage H1 is motto soup with no service/city/proof, mobile above-the-fold has no way to call, ~40 page titles carry a doubled brand suffix, and the maps link/schema geo point at the old office. The best trust assets (client logos, 24-hour deployment, pricing transparency) exist but are buried far from decision points, and CTA labeling is fragmented across seven variants with 'Free' missing from the most-seen ones.


## CRITICAL (7)

### C1. Global keyboard focus indicator is nearly invisible on the dark theme (1.3-1.7:1)
**Area:** Accessibility · **Where:** `src/app/globals.css:201` · **Effort:** quick · **Verified:** confirmed

Almost no component defines its own focus style, so every link, button, and nav item on the site relies on this outline. At 1.3-1.7:1 against the dark surfaces (WCAG 1.4.11 requires 3:1), sighted keyboard users effectively cannot see where focus is anywhere on the site — a site-wide keyboard operability failure (WCAG 2.4.7).

**Evidence:** :focus-visible { outline: 2px solid var(--color-accent) } where --color-accent = #1a3a6b (globals.css:24). Computed contrast: 1.72:1 on nav/footer #040d1e, 1.44:1 on section surface #0e2138, 1.28:1 on cards #16294a. The dark-surface override (globals.css:668-678) darkened every surface but the focus color was never updated; compiled CSS on stratton-v3.vercel.app confirms `:focus-visible{outline:2px solid var(--color-accent)...}`.

**Fix:** Change the outline to a light color that clears 3:1 on all dark surfaces, e.g. `:focus-visible { outline: 2px solid var(--on-dark-accent); }` (#6f9bd8 = 4.4:1 on #040d1e) or #f1f4f8, and add `outline-offset: 3px` as today. Verify on nav (#040d1e), sections (#0e2138), and cards (#16294a).

### C2. 3.3-second fake 'Establishing secure session' preloader blocks every fresh page load
**Area:** Conversion & Copy · **Where:** `src/components/layout/Preloader.tsx:30-43` · **Effort:** quick · **Verified:** confirmed

Every cold entry — including someone clicking a Google result or ad straight to /contact or /services/patrol — stares at a 3.25-second fake loading animation with scrolling locked before seeing any value prop or phone number. The skip affordance is invisible (the whole screen is a button with only aria-label="Skip intro"; no visible 'Skip' text). For a lead-gen site whose entire job is a call or quote request, an artificial 3+ second gate on first paint directly inflates bounce, especially on paid/mobile traffic.

**Evidence:** Preloader.tsx:30 `const DURATION = 2100;` plus timeouts at 2150/2450/3250ms with `document.body.style.overflow = "hidden"`; mounted globally in SiteChrome.tsx (`<Preloader />` wraps all non-/keystatic routes via app/layout.tsx:98). Live homepage HTML confirms overlay text renders first: "Establishing secure session … SECURE CHANNEL 0 %". Status strings include "Verifying credentials · PPO 122163", "Encrypting channel", "Perimeter secured".

**Fix:** Cap the sequence at ~800ms, run it only on the homepage, and persist a sessionStorage flag so it plays once per session. Add a visible 'Skip' label (e.g. 'Tap to skip'). Better: reserve the theater for '/' only and let deep-linked pages (contact, services, service-areas) render instantly.

### C3. Preloader forces a ~3.25s full-screen overlay on every page entry, and never exits if JS fails
**Area:** Performance · **Where:** `src/components/layout/Preloader.tsx:38` · **Effort:** quick · **Verified:** confirmed

Every visitor landing from Google (the most common entry) waits ~3.3 seconds plus JS download/hydration before any real content is visible. Lighthouse will measure LCP behind this overlay, putting mobile LCP well past the 2.5s 'good' threshold — this alone guarantees failing Core Web Vitals, which feeds into rankings. It is mounted in SiteChrome (layout-level) so it fires on every hard navigation, including deep links to /services/* and /contact where prospects arrive from ads/search.

**Evidence:** Preloader.tsx:30-43 hard-codes DURATION=2100 with setTimeout(...,2450) for exit and setTimeout(...,3250) for done. The overlay (fixed inset-0 z-[120], line 69) is present in the served SSR HTML (grep 'z-\[120\]' on live homepage HTML = 1 match; 'Establishing secure session' present), so it covers content from first paint. There is no sessionStorage/returning-visitor gate — it plays on every full page load, and phase only advances via client timers, so with JS disabled/failed the site stays permanently covered.

**Fix:** Gate it to once per session: check sessionStorage.getItem('preloaded') in an effect and render null (phase='done') on repeat loads; set the flag on first play. Also cut the sequence to <=1000ms (DURATION 700, exit 800, done 1400) and consider rendering it only after mount (return null during SSR) so first paint shows the real hero and the overlay never traps no-JS users.

**Verifier corrections:** Independently reproduced all evidence. Code (Preloader.tsx:30-43) hard-codes DURATION=2100 with exit at 2450ms and done at 3250ms; overlay is fixed inset-0 z-[120] with opaque panels (line 69); phase advances only via client timers/clicks, so no-JS users are permanently covered; no sessionStorage gate exists, and SiteChrome (mounted in root layout.tsx:98) fires it on every route except /keystatic. Curled the live site: SSR HTML on /, /contact, and /services all contain the opacity:1 overlay and 'Establishing secure session' text, confirming it covers content from first paint on deep links. Minor corrections: content starts revealing ~2.6s (not 3.25s) as the doors slide, though full clearance is 3.25s plus JS download/hydration; the finding omits that the overlay is a click-anywhere skip button and that prefers-reduced-motion bypasses it (both JS-dependent, so no help for first paint or no-JS); and the Lighthouse LCP mechanism is imprecise (Chrome LCP ignores z-index occlusion) but the conclusion holds because the hero content itself SSRs at opacity:0 pending framer-motion hydration, so no real content paints before ~2.5s+ regardless. Severity and fix are appropriate.

### H2. Mobile nav drawer: no focus trap, no Escape, focus never moved or restored, background stays tabbable
**Area:** Accessibility · **Where:** `src/components/layout/Navigation.tsx:221` · **Effort:** moderate · **Verified:** confirmed

Keyboard users who open the drawer keep tabbing into the visually hidden page beneath it (WCAG 2.4.3 focus order, 1.3.2). Escape does nothing (expected dialog behavior). When the drawer closes via a link, focus is dropped to <body> instead of returning to the hamburger button. Screen-reader users can wander into background content while the drawer covers it.

**Evidence:** The drawer (Navigation.tsx:221-353) is a plain motion.div with `fixed inset-0 z-50`. ### H3. Preloader: focusable 'Skip intro' button inside an aria-hidden overlay, with focus outline suppressed
**Area:** Accessibility · **Where:** `src/components/layout/Preloader.tsx:73` · **Effort:** quick · **Verified:** partially-true

Focusable content inside aria-hidden is a WCAG 4.1.2 failure: the first Tab press lands on a button that screen readers cannot perceive or name (a 'ghost' focus stop). focus:outline-none additionally removes the visible focus indicator (2.4.7), so sighted keyboard users also get no cue that pressing Enter skips the intro. Focus is also not trapped, so Tab reaches the covered page during the animation.

**Evidence:** Wrapper motion.div has aria-hidden="true" (Preloader.tsx:73) yet contains <motion.button ... aria-label="Skip intro" className="... focus:outline-none"> (:90-97). The overlay blocks the page for ~3.25s on every full load (:38-43) and sets body overflow hidden (:27).

**Fix:** Remove aria-hidden from the wrapper (or move it to the decorative layers only), keep the button exposed with its label, and delete focus:outline-none so the global focus ring applies. Optionally announce via aria-live="polite" that content is loading, or better: skip the preloader entirely for keyboard/AT by making it non-blocking (pointer-events only overlay).

**Verifier corrections:** Core violation confirmed by reading Preloader.tsx and the live site's SSR HTML: the aria-hidden="true" wrapper (line 73) contains the focusable Skip-intro button (lines 90-97), which is the first focusable element on the page (Preloader is first in SiteChrome, first in <body>, no skip link), the overlay blocks ~3.25s (lines 38-43), sets body overflow hidden (line 27), and has no focus trap — a real WCAG 4.1.2 / axe aria-hidden-focus failure. However, the 2.4.7 detail and part of the fix are wrong: in the live compiled CSS, .focus\:outline-none:focus{outline-style:none} is inside @layer utilities while the site's global :focus-visible{outline:2px solid ...} rule (globals.css:201) is UNLAYERED, and unlayered declarations win the cascade — so focus:outline-none does not actually suppress the keyboard focus ring, and deleting it (as the fix suggests) changes nothing. The missing visible focus cue is real but caused by geometry: the button spans the full viewport (absolute inset-0 in a fixed inset-0 overflow-hidden wrapper), so the 2px ring with outline-offset:3px renders entirely off-screen. Severity is also slightly overstated: the ghost focus stop is transient (3.25s window) and prefers-reduced-motion users bypass the preloader entirely (lines 22-26). The primary fix (remove aria-hidden from the wrapper or scope it to decorative layers) remains correct.

### H4. Client-logo marquee auto-scrolls indefinitely with hover-only pause — no mechanism for touch or keyboard users (WCAG 2.2.2, Level A)
**Area:** Accessibility · **Where:** `src/app/globals.css:124` · **Effort:** quick · **Verified:** confirmed
